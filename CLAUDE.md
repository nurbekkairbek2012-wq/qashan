## Один пользователь — одна боль (правило регламента, TVEP)

**Пользователь:** студент 17–20 лет с первым доходом (стипендия / подработка).

**Боль:** берёт рассрочку «0-0-4». Потом вторую в другом магазине, потом третью. Каждая
по отдельности выглядит безобидно — «всего 15 000 в месяц». Совокупной картины не видит
никто, потому что каждая рассрочка живёт в своём приложении. В какой-то месяц платежи
молча съедают весь доход.

**USP:** показываем не «сколько ты должен» (это показывает любой банк), а **когда именно
ты уйдёшь в минус** — и **что будет, если взять ещё одну рассрочку, ДО того как ты её взял**.

⚠️ Расширять на «финансовую грамотность вообще» — НЕЛЬЗЯ. Жюри проверяет глубину одной
находки, а не широту охвата. Любая фича, не служащая этой боли, — вон.

## Ключевая находка (и что в ней проверено, а что нет)

**Устойчивый факт — на нём держится боль:** совокупная нагрузка невидима. Три рассрочки
в трёх приложениях, никто не показывает их сумму и месяц ухода в минус. Это верно
независимо от любых допущений про проценты.

**Алгоритмическая модель:** считаем **эффективную ставку через IRR** (внутренняя норма
доходности) численным методом — своя реализация, не библиотека. Плюс график амортизации
и помесячная симуляция денежного потока → первый месяц с отрицательным балансом.
Это и есть «математическая/алгоритмическая модель решения» для TVEP (стр. 3 паспорта).

⚠️ **НЕПРОВЕРЕННАЯ ГИПОТЕЗА — не подавать как факт:** «в рассрочке 0-0-4 зашита скрытая
наценка». В Kaspi Red комиссию платит продавец, и скрытого процента для покупателя может
не быть вовсе. Если заявить это на защите бездоказательно — первый же член жюри, покупавший
в Kaspi, развалит USP.

Гипотезу **проверяем данными** (реальные пары цен, см. ниже), а не подтверждаем задним числом.
- Подтвердится → сильная находка, IRR показывает скрытый процент.
- Не подтвердится → честно говорим это на защите; IRR остаётся для Kaspi Кредит на 12–24 мес,
  где процент реальный и явный. Боль (невидимая совокупная нагрузка) не страдает ни в одном случае.

Честно показанная опровергнутая гипотеза — это глубина исследования (критерий №1),
а не провал. Подгонка под красивый вывод — это провал.

## Стек

- **Frontend:** React + Vite + Tailwind
- **БД / Auth / Storage:** Supabase (PostgreSQL, RLS)
- **Серверная логика:** Supabase Edge Functions (Deno/TS) — расчёты и AI-парсинг.
  Ключи API на клиент НЕ попадают (критерий «безопасность» в финале).
- **AI:** Claude API — vision-парсинг скриншота/СМС о рассрочке → структурированный JSON.
  Кастомная цепочка: извлечение → нормализация → восстановление графика платежей.
  НЕ «пустая обёртка над API» — это прямо наказывается по критерию №2.

## Данные — только реальные (решение от 17.07.2026)

Синтетику **не используем**. Kaspi публичного API не имеет, но реальные данные достижимы:

| Источник | Что берём | Зачем |
|---|---|---|
| **CustDev-собеседник** | его настоящие рассрочки: суммы, сроки, магазины | демо крутится на живом кейсе |
| **Публичные страницы товаров** | 10–20 пар «цена за наличные / цена в рассрочку», **руками** | проверка гипотезы о наценке |
| **Нацбанк РК** | базовая ставка | есть с чем сравнить эффективную |
| **stat.gov.kz** | медианный доход молодёжи по регионам | обоснование масштаба в питче |

**Два жёстких ограничения:**

1. **Массовый автоскрейпинг Kaspi — НЕ делаем.** Против ToS, ломается на редизайне, и на защите
   это вопрос «а вы имели право?». Только ручной сбор небольшого набора — законно и достаточно.
2. **Финансовые данные живого человека** — только с явного согласия и обезличенно.
   В демо, видео и репозитории: никаких имён, номеров карт, реальных телефонов.
   Иначе на публичное видео уедут чужие долги.

CustDev — задача пользователя, не Claude. **Интервью не выдумывать ни при каких условиях:**
это обман жюри, и разваливается первым вопросом «с кем конкретно вы говорили?».
Если данных не хватило — честно сказать в README и на защите (регламент это прямо разрешает, стр. 2).

## Устройство кода (кратко)

```
frontend/src/
  core/       — чистые функции без React, всё под тестами:
    dates, schedule, irr, simulate, format, normalize, verdict
  lib/        — supabase (клиент), repository (localStorage ⇄ Supabase)
  context/    — AuthContext (вход необязателен)
  components/ — Dashboard, CashflowChart, WhatIfSimulator,
                InstallmentForm/List, AuthPanel, ScreenshotUpload
supabase/
  migrations/ — схема + RLS
  functions/  — parse-installment (vision на Gemini)
```

Принцип: расчёты и симулятор работают без входа (localStorage). Вход нужен
для синхронизации и парсинга скриншотов. Источник данных прячет repository —
App не ветвится на «есть база или нет».
