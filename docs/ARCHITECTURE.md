# Qaryz — архитектура и модель расчёта

Документ проектирования. Хакатон Tech Vision 2026, направление Digital Economy & Future Tech,
проблемная зона 11 (Финтех).

---

## 1. Боль и границы продукта

**Пользователь:** студент 17–20 лет с первым доходом (стипендия / подработка).

**Боль:** берёт рассрочку «0-0-4», потом вторую в другом магазине, потом третью. Каждая
по отдельности выглядит безобидно — «всего 15 000 в месяц». Совокупной картины не видит
никто: каждая рассрочка живёт в своём приложении. В какой-то месяц платежи молча съедают
весь доход.

**Что делает Qaryz:** показывает не «сколько ты должен» (это показывает любой банк), а
**в каком месяце ты уйдёшь в минус** — и **что будет, если взять ещё одну рассрочку,
до того как ты её взял**.

**Чего Qaryz НЕ делает** (правило регламента «один пользователь — одна боль»):
не учит финансовой грамотности вообще, не считает инвестиции, не ловит мошенников,
не ведёт бюджет по категориям. Любая фича, не служащая этой боли, отклоняется.

## 2. Схема данных (PostgreSQL / Supabase)

Все таблицы под RLS: пользователь видит только свои строки (`user_id = auth.uid()`).
Исключение — `price_observations`, это исследовательские данные, читаемые всеми.

| `price_installment` | numeric | цена товара в рассрочку |

## 3. Модель расчёта

Три независимых блока. Каждый — чистая функция без побочных эффектов, покрыт тестами.

### 3.1. Построение графика платежей

`buildSchedule(installment) → Payment[]`

Из суммы, срока и даты первого платежа разворачиваем помесячный график.
Тонкость: если день платежа 31-е, а в месяце 30 дней — платёж переносится на последний
день месяца (а не на 1-е следующего). Это же правило у банков.

### 3.2. Эффективная ставка через IRR

Ключевая математика проекта. Модель денежного потока **с точки зрения покупателя**:

```
t=0:      +L,  где L = price_cash − down_payment
          (ты получил товар, за который иначе отдал бы price_cash наличными;
           down_payment уже уплачен, поэтому вычитается)

t=1..n:   −monthly_payment   (ежемесячные платежи)
```

IRR — это ставка `r`, при которой чистая приведённая стоимость равна нулю:

```
NPV(r) = Σ ( CF_t / (1 + r)^t ) = 0,   t = 0..n
```

`r` — месячная ставка. Годовая эффективная: `(1 + r)^12 − 1`.

**Почему это честная модель, а не подгонка:**
- Если `price_installment == price_cash`, то сумма платежей равна `L`, и IRR выходит **ровно 0**.
  Рассрочка действительно беспроцентная — модель это подтверждает, а не выдумывает процент.
- Если `price_installment > price_cash`, IRR > 0 и показывает реальную стоимость наценки
  в годовых — то, что нигде не написано.
- Если `price_cash` неизвестна — **ставку не считаем**. Никаких допущений.

**Численный метод: бисекция.**

Аналитического решения для полинома степени n нет. Берём бисекцию на отрезке `r ∈ [−0.99, 10]`:
NPV монотонно убывает по `r`, поэтому при смене знака на концах корень существует и
единственен. Делим отрезок пополам, пока не сойдёмся до `1e-10` или 200 итераций.

Почему бисекция, а не Ньютон: Ньютон быстрее, но может разойтись при плохом начальном
приближении и требует производной. Бисекция сходится **всегда** при смене знака.
На наших размерах (n ≤ 24) разница в скорости неощутима, а надёжность важнее —
демо на сцене не должно упасть.

### 3.3. Симуляция денежного потока

`simulateCashFlow(profile, installments, horizonMonths) → MonthState[]`

Помесячно вперёд на горизонт (по умолчанию 12):

```
balance[m] = balance[m−1] + monthly_income − Σ(платежи, попадающие в месяц m)
```

Возвращает массив состояний по месяцам и **первый месяц с отрицательным балансом** —
главный ответ продукта. Плюс коэффициент нагрузки:

```
burden[m] = Σ(платежи месяца m) / monthly_income
```

Пороги (обоснование в README со ссылкой на практику долговой нагрузки):
`< 0.30` — safe · `0.30–0.50` — warn · `> 0.50` — danger.
Эти же три уровня — цветовая шкала интерфейса.

### 3.4. Сценарий «а что если»

`simulateWithDraft(profile, installments, draft) → { before, after, delta }`

Тот же симулятор, но к списку добавляется черновая рассрочка. Возвращаем **разницу**:
насколько сдвинулся месяц ухода в минус. Это и есть превентивная ценность продукта —
ответ до покупки, а не после.

## 4. Компоненты системы

```
┌─────────────────────────────────────────────┐
│  Frontend — React + Vite + Tailwind         │
│  · ввод рассрочки (руками или скриншотом)   │
│  · дашборд: нагрузка, месяц ухода в минус   │
│  · симулятор «что если»                     │
└───────────────┬─────────────────────────────┘
                │ supabase-js (anon key, RLS)
┌───────────────▼─────────────────────────────┐
│  Supabase                                   │
│  · PostgreSQL + RLS                         │
│  · Auth (email)                             │
│  · Storage (скриншоты рассрочек)            │
└───────────────┬─────────────────────────────┘
                │
┌───────────────▼─────────────────────────────┐
│  Edge Function (Deno) — parse-installment   │
│  · принимает скриншот                       │
│  · Claude API (vision) → структура          │
│  · нормализация + валидация                 │
│  · ключ API живёт ТОЛЬКО здесь              │
└─────────────────────────────────────────────┘
```

**Почему ядро расчётов на клиенте, а не на сервере.** Расчёт — чистая функция от данных,
которые пользователь и так видит. Прятать нечего, а мгновенный отклик в симуляторе
«что если» — это и есть продукт. На сервере то, что прятать **нужно**: ключ Claude API.

**Почему AI-парсинг на сервере.** Ключ API на клиенте — это ключ, украденный из бандла
в первый день. Edge Function его не отдаёт.

## 5. Роль ИИ — и почему это не обёртка

Критерий №2 прямо наказывает «пустые обёртки над API». Цепочка `parse-installment`:

1. **Vision-извлечение** — скриншот → черновой JSON.
2. **Нормализация** — «15 000 ₸», «15000₸», «15 тыс» → `15000`. Даты в ISO. Детерминированный
   код, не модель.
3. **Валидация инварианта** — `monthly_payment × term_months ≈ price_installment − down_payment`.
   Не сходится → не сохраняем молча, а показываем расхождение и просим подтвердить.
4. **Расчёт** — детерминированный, моделью не считаем. Модель **распознаёт**, арифметику
   делает код.

Разделение принципиальное: **LLM извлекает, код считает.** Языковая модель ошибается в
арифметике, и доверять ей деньги нельзя. Это же ответ на вопрос жюри «почему ИИ здесь
вообще нужен?» — он нужен ровно там, где надо прочитать неструктурированный скриншот,
и нигде больше.

## 7. Открытый вопрос (проверяется данными, не допущением)

Гипотеза: «в рассрочке 0-0-4 зашита скрытая наценка».

Против неё: в Kaspi Red комиссию платит продавец, и для покупателя переплаты может
не быть вовсе.

Решается таблицей `price_observations`. Подтвердится — сильная находка. Не подтвердится —
честно фиксируем в README и на защите; IRR остаётся для Kaspi Кредит на 12–24 месяца, где
процент реальный. **Боль от этого не зависит:** невидимая совокупная нагрузка существует
при любой ставке, хоть при нулевой.
