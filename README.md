## Проблема

**Один пользователь:** студент 17–20 лет с первым доходом — стипендия или подработка.

**Одна боль:** он берёт рассрочку «0-0-4». Потом вторую в другом магазине. Потом третью.
Каждая по отдельности выглядит безобидно — «всего 15 000 в месяц». Но совокупной картины
не видит никто: каждая рассрочка живёт в своём приложении. В какой-то месяц платежи молча
съедают весь доход.

Банк показывает, **сколько ты должен**. Никто не показывает, **когда ты не потянешь**.

## Решение

Qaryz отвечает на три вопроса:

1. **В каком месяце ты уйдёшь в минус** — помесячная симуляция денежного потока.
2. **Что будет, если взять ещё одну** — симулятор до покупки, а не разбор после.
3. **Что если не брать** — план накопления: тот же платёж, но себе, а не банку.

Плюс считает **реальную эффективную ставку через IRR**: если в рассрочке зашита наценка,
она превращается в понятное «это X% годовых». Если наценки нет — модель честно показывает 0.

Третий вопрос появился потому, что диагноз без выхода бесполезен: человек, которому
сказали «в ноябре не потянешь», всё равно возьмёт рассрочку, если другого пути ему
не показали. Цену альтернативы — ждать N месяцев — называем прямо, а не прячем в сноску.

## Роль ИИ

**LLM извлекает — код считает.** Модель распознаёт скриншот и отдаёт черновые
*строки*. Числа, даты и вся арифметика (график платежей, IRR, симуляция) —
детерминированный код, покрытый тестами. Языковой модели деньги считать не даём:
она ошибается в арифметике.

Цепочка: скриншот → Edge Function (Gemini) → сырые строки → `normalize.js`
(числа/даты) → валидация инварианта `платёж × срок ≈ сумма` → предзаполнение
формы. Расхождение не проглатывается молча — показывается человеку.

Это не обёртка над API: модель отвечает только за распознавание текста с
картинки, а всё, что влияет на вывод о деньгах, живёт в `core/` под тестами.

## Данные

**Синтетических данных в проекте нет.** Мы принципиально не подставляем выдуманные
цифры вместо несобранных — вместо этого честно показываем, что собрано, а что нет.

### Статус сбора на 19 июля 2026

| Что | Статус |
|---|---|
| Пары цен «наличные / рассрочка» | **1**, и та из публикации, а не с витрины — [`docs/price_observations.csv`](docs/price_observations.csv). Собственных наблюдений пока нет |
| Кабинетное исследование наценки | **проведено** — правила площадок и позиция госорганов, см. «Открытый вопрос» |
| CustDev-интервью | в репозитории пока не зафиксированы; методика — [`docs/CUSTDEV.md`](docs/CUSTDEV.md) |

Кнопка «Показать на примере» в интерфейсе грузит **явно помеченные демо-данные** —
они существуют только для показа механики и нигде не выдаются за реальные.

Расчётное ядро от этого не зависит: оно считает те цифры, которые ввёл человек.

### Откуда берём данные, когда они есть

| Источник | Что берём |
|---|---|
| CustDev-интервью | настоящие рассрочки — с согласия, обезличенно |
| Публичные страницы товаров | пары «цена за наличные / цена в рассрочку», собранные **вручную** |
| Нацбанк РК | базовая ставка — база для сравнения |
| stat.gov.kz | медианный доход молодёжи по регионам |

Массового автоскрейпинга не ведём. Персональные финансовые данные — только с согласия
и обезличенно: в демо нет реальных имён и номеров.

### Что из списка организаторов мы НЕ используем — и почему

Оргкомитет опубликовал общий список открытых источников на все пять проблемных зон
направления. Осознанно берём из него два и не берём остальные:

| Источник | Решение |
|---|---|
| stat.gov.kz, data.egov.kz | **берём** — медианный доход молодёжи, обоснование масштаба |
| Локальные СМИ (kursiv.media, zakon.kz…) | **берём** — свежие цифры по рассрочкам для контекста CustDev |
| Kazhydromet, IQAir, air.org.kz | не берём — качество воздуха к долговой нагрузке отношения не имеет |
| KOREM (электроэнергия) | не берём — то же |
| OpenStreetMap, погодные API | не берём — у нас нет географической составляющей |

Это не пропуск, а следствие правила «один пользователь — одна боль»: данные, которые
не служат вопросу «когда студент уйдёт в минус», в проект не попадают. Подключить
их было бы легко и выглядело бы солиднее в списке технологий — но это ровно та
широта охвата вместо глубины, за которую снимают баллы.

## Открытый вопрос

Гипотеза «в рассрочке 0-0-4 зашита скрытая наценка» проверялась — и **в исходном виде
не подтвердилась**. Кабинетное исследование (19.07.2026) показало, что вопрос был
поставлен неверно: всё зависит от того, **кто финансирует рассрочку**.

| Кто даёт рассрочку | Наценка для покупателя |
|---|---|
| **Банк** (Kaspi Red, Technodom, маркетплейсы с БВУ) | запрещена правилами: «цена должна быть одинаковой вне зависимости от способа оплаты» ([Kaspi](https://guide.kaspi.kz/client/ru/red/shopping/general/q15451)). Комиссию платит продавец |
| **Сам продавец** | легальна с 10.10.2025 — Минторг разрешил двойные ценники. Вице-премьер С. Жумангарин оценивает переплату в **15–20%** |

Пример из прессы: iPhone 16 — **479 000 ₸** при полной оплате против **599 000 ₸**
в рассрочку ([kzvesti.kz, 08.11.2025](https://kzvesti.kz/newspaper-articles/nalichnymi-ili-v-rassrochku-cena-dolzhna-byt-ponjatnoj-146325/)).

**Что мы поменяли по итогам.** Мы больше не утверждаем «Kaspi прячет процент» — это
было бы неправдой, и первый же член жюри, покупавший в Kaspi, нас бы поправил.
Формулировка теперь такая: наценка зависит от источника финансирования, покупатель
этой разницы не видит, и поэтому поле «цена за наличные» в интерфейсе заполняет он сам —
мы не решаем за него, какой случай у него на руках.

⚠️ Всё перечисленное — **вторичные источники** (СМИ и правила площадок), а не наши
собственные наблюдения. Своих пар цен собрано **1** (и та из публикации, помечена в
[CSV](docs/price_observations.csv) как вторичная). Разбор — [docs/PRICE_RESEARCH.md](docs/PRICE_RESEARCH.md).

**Боль от этого не пострадала:** невидимая совокупная нагрузка существует при любой
ставке, хоть при нулевой. Именно её продукт и показывает.

Боль от этого не зависит: невидимая совокупная нагрузка существует при любой ставке,
хоть при нулевой.

## Что работает

- **Ввод рассрочек** — вручную или загрузкой скриншота (распознаётся автоматически).
- **Дашборд** — месяц, в котором платежи перевесят доход; помесячный график; нагрузка.
- **Совокупная картина** — остаток долга в месяцах твоего дохода, месяц последнего
  платежа, переплата. Ни одно банковское приложение этого не показывает: каждое
  видит только свою рассрочку.
- **Разбивка по рассрочкам** — составной график: видно не «65 000 в ноябре», а какая
  именно покупка тебя топит.
- **Симулятор «а что если»** — вводишь цену будущей покупки и видишь, на сколько
  приближается уход в минус. До покупки, а не после.
- **План накопления** — сколько откладывать, когда купишь без долга, сколько
  сэкономишь. Считается без инфляции и процента на вклад, о чём сказано в интерфейсе:
  это верхняя граница экономии, а не обещание.
- **Реальная ставка** по каждой рассрочке через IRR (где известна цена за наличные).
- **Вход необязателен**: расчёты работают на localStorage без аккаунта. Вход нужен
  для синхронизации между устройствами и разбора скриншотов.

## Локальный запуск

Требуется Node.js 20+ (разрабатывалось на 22).

```bash
git clone https://github.com/nurbekkairbek2012-wq/qashan.git
cd qashan/frontend
cp .env.example .env      # вписать свои ключи Supabase (или оставить пустым —
                          # приложение поднимется в localStorage-режиме)
npm install
npm run dev               # http://localhost:5173
```

Команды:

```bash
npm run dev       # dev-сервер
npm run build     # production-сборка в dist/
npm run preview   # предпросмотр сборки
npm run test      # vitest — ядро расчётов, нормализация, рендер (82 теста)
npm run lint      # oxlint
```

Переменные окружения — см. `frontend/.env.example`. Ключи в репозиторий не
коммитятся; `anon key` не секретный (его закрывает RLS), ключ Gemini живёт
в секретах Supabase на сервере.

### Supabase (опционально — без него работает localStorage)

```bash
# схема + RLS: Supabase Dashboard → SQL Editor → выполнить
supabase/migrations/0001_init.sql

# распознавание скриншотов:
npx supabase functions deploy parse-installment
# секрет GEMINI_API_KEY задаётся в Dashboard → Edge Functions → Secrets
```
